# 📰 AI News Insight Bot (Dual-Knowledge RAG Architecture)

最新のAIニュースと基礎知識を組み合わせて回答する、**RAGベースのニュース解説アプリ**です。  
ITリテラシーが高くないユーザー（自身の家族）でも迷わず使えることを目的に、**チャット入力中心のシンプルなUI**と、**時事ニュース＋静的知識を組み合わせた二層RAG構成**で設計しました。

---

## 🎯 プロジェクト概要

このアプリは、**「最新ニュースには強いが、基礎知識の質問に弱い」**という一般的なニュースRAGの弱点を補うために開発しました。

通常のニュースRAGでは、直近の記事だけを検索対象にすると、たとえば「Sakana AIとは？」のような基礎的な質問に対して、その週に関連記事がない場合は十分に回答できないことがあります。

そこで本アプリでは、以下の2種類の知識ソースを分けて保持する構成にしています。

- **最新ニューステーブル**：一定期間で更新・削除される時事情報
- **基礎知識テーブル**：AI用語や企業情報などを永続保持する静的ナレッジ

この2層構造により、**時事性の高い質問**にも**基礎的な用語質問**にも対応しやすいRAG構成を実現しています。

---

## 🧩 解決した課題

### 1. 高齢ユーザーでも使いやすいUI設計

#### 課題
当初は情報量の多いダッシュボード形式で実装していましたが、家族に使ってもらったところ、  
**「どこを見ればいいか分からない」**というフィードバックがありました。

#### 対応
- Streamlitの不要なヘッダー / フッター / メニュー要素をCSSで非表示化
- UIを**チャット入力欄中心**に再設計
- 「最初に何をすればいいか」が直感的に分かる導線に変更

#### 学び
機能を足すよりも、**ユーザーの認知負荷を減らす“引き算のUI”**の方が重要な場面があると実感しました。

---

### 2. ニュースRAGの弱点を補う二層知識構造

#### 課題
ニュース記事のみを検索対象にした場合、  
- 最新ニュースの要約には強い  
- 一方で、基礎知識を問う質問に弱い  

という問題がありました。

#### 対応
SQLite内に以下の2テーブルを分離して保持しています。

- `news`：最新ニュース（期限付きで更新）
- `knowledge`：AI用語・企業・基礎知識（永続保持）

検索時は、  
1. ユーザー質問をEmbedding化  
2. 両テーブルの候補データを取得  
3. Pandasで結合  
4. Numpyでコサイン類似度を計算  
5. 上位コンテキストをGeminiに渡して回答生成  

という流れで、**時事性と基礎知識を両立**しています。

---

### 3. 手動更新をなくす日次バッチ自動化

#### 課題
ニュース収集・ベクトル化・古いデータの削除を手動で行うと、運用コストが高く継続しづらい。

#### 対応
GitHub Actionsを使い、以下を**日次で自動実行**するようにしました。

- RSSフィードから最新ニュース取得
- 記事本文の抽出
- Gemini Embedding APIでベクトル化
- SQLite DBの更新
- 古いニュースデータの削除
- 更新済みDBをリポジトリへ反映

#### 学び
アプリ本体だけでなく、**「継続して動かす仕組み」まで含めて設計することの重要性**を学びました。

## 🏗 アーキテクチャ概要

### 1. データ収集バッチ（GitHub Actions）
定期的にRSSフィードからAIニュースを取得し、本文抽出・Embedding生成を行った上で `news.db` を更新します。

### 2. 静的ナレッジ投入
`seed_knowledge.py` により、AI用語・企業情報などの基礎知識を `knowledge` テーブルへ登録します。

### 3. 推論フロー（RAG）
1. ユーザーの質問をEmbedding化  
2. `news` / `knowledge` の両テーブルから候補データを取得  
3. 類似度計算で関連度の高いコンテキストを抽出  
4. Geminiへコンテキスト付きでプロンプト送信  
5. 回答生成  

### 4. RAG特有の精度低下と文脈切れへの対応

#### 課題
長いニュース記事をそのまま1つのベクトルにすると、回答に必要なピンポイントな情報が薄もれて精度が落ちる問題がありました。また、1問1答形式では「それはどういう意味？」といった指示語を含む連続した質問（深掘り）に対応できませんでした。

#### 対応
- **Chunking（チャンキング）の導入**：記事本文を一定文字数（Overlapあり）で分割してからベクトル化し、DBに保存。Geminiに渡すコンテキストの純度を高めました。
- **Chat Memory（文脈記憶）の実装**：Streamlitの `session_state` を活用し、直近の会話履歴をプロンプトに結合。文脈を理解したマルチターン会話を実現しました。

#### 学び
LLMの性能を引き出すには、単にデータを渡すだけでなく「データの解像度（Chunking）」と「プロンプトの設計（Memory）」が不可欠であることを実体験として学びました。

## 🔍 実装上の工夫

### ベクトルDBを使わず、SQLite + Numpyで検索基盤を自作
小規模データを前提に、あえて外部ベクトルDBは使わず、以下の構成で実装しました。

- SQLiteでデータ保持
- Pandasで検索対象データを整形
- Numpyでベクトル演算（コサイン類似度）
- 上位k件をRAGコンテキストとして採用

この構成にした理由は、  
**小規模用途では十分に成立すること**と、  
**RAGの検索ロジックをブラックボックス化せず、自分で理解しながら実装したかったため**です。

---

## 💡 このプロジェクトで学んだこと

- **RAGは「つなぐだけ」ではなく、検索対象の設計が重要**
- **時事情報と静的知識は、同じデータとして扱わない方がよいケースがある**
- **ユーザーにとって使いやすいUIは、情報量より導線の明確さが重要**
- **個人開発でも、GitHub Actionsで運用自動化まで作ると継続しやすい**
- **小規模RAGでは、ベクトルDBを使わなくても構造理解を優先した実装ができる**

## 🚀 ローカル環境での動かし方

```bash
# リポジトリをクローン
git clone https://github.com/your-username/news-rag-app.git
cd news-rag-app

# 依存パッケージをインストール
pip install -r requirements.txt

# APIキーを設定（.envファイルを作成）
echo "GEMINI_API_KEY=your_api_key" > .env

# 基礎知識データを投入（初回のみ）
python seed_knowledge.py

# ニュースデータを取得（任意）
python check.py

# アプリ起動
streamlit run web.py
